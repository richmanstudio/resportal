# Architecture

РЕСПОРТАЛ строится как multi-tenant SaaS. Все рабочие данные привязаны к `organizationId`; API получает активную организацию из заголовка `x-organization-id` и проверяет членство пользователя перед доступом к данным.

Backend остается модульным монолитом. Это проще для MVP, дешевле в эксплуатации и достаточно для целевой нагрузки: 100 организаций, 1000 пользователей, 10 000 дел и 100 000 документов.

## MVP Scope

РЕСПОРТАЛ ориентирован на частнопрактикующих юристов и небольшие юридические фирмы. MVP должен закрывать ежедневный рабочий цикл: регистрация, организация, клиенты, дела, стороны дела, задачи, процессуальные сроки, календарь, документы, генерация DOCX, сотрудники и оплата.

В MVP сознательно не входят AI-функции, интеграции с судебными системами, мобильное приложение, микросервисы и сложная BI-аналитика. Эти направления не должны влиять на архитектуру первой версии.

## Product Flow

```text
User
  -> Auth
  -> Organization
  -> Clients
  -> Legal Cases
  -> Case Parties
  -> Tasks / Deadlines / Events
  -> Documents / DOCX Generation
  -> Members / Roles
  -> Billing
```

Основные модули:

- Auth: регистрация, вход, refresh, logout, текущий пользователь.
- Organizations: рабочие пространства и роли.
- Clients: карточки доверителей и контрагентов.
- Cases: судебные дела и стороны дела.
- Case Parties: участники дела, представители, суд и иные стороны.
- Documents: загрузка, хранение, скачивание и генерация DOCX.
- Deadlines: процессуальные сроки и контроль исполнения.
- Tasks: рабочие задачи по делам и без привязки к делу.
- Events: календарные события, заседания, звонки и встречи.
- Billing: тарифы, лимиты и оплата через YooKassa.

## Security Baseline

- Все рабочие данные фильтруются по `organizationId`.
- Все tenant-маршруты требуют `Authorization: Bearer <token>` и `x-organization-id`.
- Роли ограничивают запись, destructive-действия и управление участниками.
- Production config валидируется при старте.
- Production не должен запускаться со слабыми JWT secrets.
- Rate limiting использует Redis при наличии `REDIS_URL` и memory fallback в локальной разработке.
- Важные действия пишутся в `AuditLog`.
